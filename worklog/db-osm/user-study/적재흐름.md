# OSM 적재 전체 흐름 (테이블 생성 → main.py 실행 → DB 적재)

`python -m loader.main` 한 줄을 실행했을 때 어떤 함수가 어떤 순서로 호출되고, 어떤 데이터가 어떻게 변환되어 어떤 SQL로 저장되는지를 코드·SQL·실 데이터 예시 단위로 정리.

---

## 0. 사전 단계 — 테이블 생성 (적재 전 1회만)

main.py를 실행하기 전에 DB 쪽에 테이블이 있어야 한다. main.py는 INSERT만 하고 테이블은 안 만든다.

### 적용 명령
```bash
docker exec -i localPostgis psql -U root -d running < sql/01_extensions.sql
docker exec -i localPostgis psql -U root -d running < sql/02_schema.sql
```

### `sql/01_extensions.sql` 실행 내용
```sql
CREATE EXTENSION IF NOT EXISTS postgis;
```
PostGIS 확장 활성화. 이미 있으면 NOTICE만 띄우고 넘어감.

### `sql/02_schema.sql` 실행 내용
```sql
CREATE TABLE IF NOT EXISTS osm_ways ( ... 16 컬럼 ... );
CREATE INDEX IF NOT EXISTS idx_osm_ways_geom        ON osm_ways USING GIST(geom);
CREATE INDEX IF NOT EXISTS idx_osm_ways_geom_geog   ON osm_ways USING GIST((geom::geography));
CREATE INDEX IF NOT EXISTS idx_osm_ways_highway     ON osm_ways(highway);

CREATE TABLE IF NOT EXISTS osm_crossings ( ... 6 컬럼 ... );
CREATE INDEX IF NOT EXISTS idx_osm_crossings_geom       ON osm_crossings USING GIST(geom);
CREATE INDEX IF NOT EXISTS idx_osm_crossings_geom_geog  ON osm_crossings USING GIST((geom::geography));
```

이 시점 DB 상태:
```
osm_ways      : 0건  (스키마만 존재)
osm_crossings : 0건
```

## 1. 진입점 — `python -m loader.main`

```python
# loader/main.py
if __name__ == "__main__":
    main()
```

`main()` 함수가 호출되며 아래 7단계를 순차 실행한다.

```
┌─────────────────────────────────────────────────────────────┐
│  1) logging.basicConfig                                      │
│  2) load_db_config()           → cfg dict                    │
│  3) get_connection(cfg)        → psycopg2 connection         │
│  4) fetch_overpass()           → Overpass JSON dict          │
│  5) parse_response(data)       → (ways[], crossings[])       │
│  6) insert_ways_batch(...)                                   │
│     insert_crossings_batch(...)                              │
│  7) _count() × 2               → 적재 결과 로그              │
│  8) finally: conn.close()                                    │
└─────────────────────────────────────────────────────────────┘
```

## 2. 단계별 상세

### 2-1. 환경변수 로드 — `load_db_config()`

**호출:** `loader/main.py` → `loader/config.py:load_db_config()`

**동작:**
1. `loader/config.py` 위치를 기준으로 프로젝트 루트(`자율/data/osm-loader/`)를 계산
2. `python-dotenv`의 `load_dotenv()`로 루트의 `.env` 파일을 읽어 `os.environ`에 주입
3. 5개 환경변수를 dict로 묶어 반환

**코드:**
```python
def load_db_config() -> dict:
    project_root = Path(__file__).resolve().parent.parent
    load_dotenv(project_root / ".env")
    return {
        "host": os.environ["POSTGRES_HOST"],
        "port": int(os.environ["POSTGRES_PORT"]),
        "dbname": os.environ["POSTGRES_DB"],
        "user": os.environ["POSTGRES_USER"],
        "password": os.environ["POSTGRES_PASSWORD"],
    }
```

**반환값 (실 데이터):**
```python
{
    "host": "localhost",
    "port": 5432,
    "dbname": "running",
    "user": "root",
    "password": "ssafy1234!"
}
```

이 dict는 곧장 `psycopg2.connect(**cfg)`에 unpack되도록 키 이름을 맞췄다.

**코드:**

def get_connection(cfg: dict):

### 2-3. Overpass API 호출 — `fetch_overpass()`

**호출:** `main.py` → `loader/overpass_client.py:fetch_overpass()`

**코드 핵심:**
```python
response = requests.post(
    "https://overpass-api.de/api/interpreter",
    data={"data": QUERY},
    headers={"User-Agent": "osm-loader/1.0 (running-app data ingestion)"},
    timeout=300,
)
response.raise_for_status()
return response.json()
```

**전송되는 쿼리 (QUERY 상수):**
```
[out:json][timeout:300];
area["name"="마포구"]["boundary"="administrative"]->.searchArea;
(
  way["highway"~"^(footway|pedestrian|path|steps|residential|...|track)$"]
     ["foot"!="no"]
     ["access"!="private"]
     ["access"!="no"]
     (area.searchArea);
  node["highway"="crossing"](area.searchArea);
);
out tags geom;
```

**Overpass 서버 처리:**
1. `area["name"="마포구"]` → 마포구 행정 경계 area 식별
2. way 필터 — highway가 footway/residential/primary 등에 해당, foot≠no, access≠private/no
3. node 필터 — highway=crossing
4. 마포구 area 안에 있는 결과만 추출
5. `out tags geom` — 모든 태그 + geometry 좌표 반환

**HTTP 응답 (JSON, 약 7초 소요, 약 6,627 element):**
```json
{
  "version": 0.6,
  "generator": "Overpass API ...",
  "elements": [
    {
      "type": "way",
      "id": 26268425,
      "geometry": [
        {"lat": 37.5386282, "lon": 126.9444697},
        {"lat": 37.538715,  "lon": 126.9443664},
        ...
      ],
      "tags": {
        "highway": "residential",
        "name": "...",
        ...
      }
    },
    {
      "type": "node",
      "id": 436804138,
      "lat": 37.5445862,
      "lon": 126.9549754,
      "tags": {
        "highway": "crossing",
        "crossing": "marked",
        ...
      }
    },
    ... (총 6,627 element: way 6,290 + node 337)
  ]
}
```

**재시도:** 실패 시 1s → 2s 백오프로 최대 3회 시도. 모두 실패하면 RuntimeError.

**반환값:** 위 JSON을 그대로 dict로 파싱한 객체.

### 2-4. 응답 파싱 — `parse_response(data)`

**호출:** `main.py` → `loader/parser.py:parse_response()`

**전체 흐름:**
```python
def parse_response(data: dict) -> tuple[list[dict], list[dict]]:
    elements = data.get("elements", [])
    ways, crossings = [], []
    skipped = 0
    for element in elements:
        if element["type"] == "way":
            parsed = parse_way(element)
            if parsed is None:
                skipped += 1
                continue
            ways.append(parsed)
        elif element["type"] == "node":
            crossings.append(parse_node(element))
    return ways, crossings
```

#### 2-4-1. `parse_way(element)` — way 1건 변환

**입력 (Overpass 형식):**
```python
{
  "type": "way",
  "id": 26268425,
  "geometry": [
    {"lat": 37.5386282, "lon": 126.9444697},
    {"lat": 37.538715,  "lon": 126.9443664}
  ],
  "tags": {"highway": "residential", "name": "..."}
}
```

**변환 로직:**
1. `geometry` 좌표가 2개 미만이면 `None` 반환 (LineString 불가)
2. `lat/lon` → `lon lat` 순서로 재배열해 WKT 문자열 생성
   - `[{"lat":37.53, "lon":126.94}, ...]` → `"LINESTRING(126.94 37.53, ...)"`
3. `tags`에서 14개 필드를 dict로 추출
4. boolean 필드(`bridge`/`tunnel`/`indoor`)는 `=="yes"`로 평가
5. `lit`만 3-state: None / True / False
6. `layer`는 `int()` 변환, 실패 시 0

**출력 (DB INSERT용 dict, 14키):**
```python
{
  "osm_id": 26268425,
  "highway": "residential",
  "name": "...",
  "geom_wkt": "LINESTRING(126.9444697 37.5386282, 126.9443664 37.538715)",
  "foot": None,
  "access": None,
  "oneway": None,
  "bridge": False,
  "tunnel": False,
  "indoor": False,
  "layer": 0,
  "surface": None,
  "lit": None,
  "sidewalk": None
}
```

#### 2-4-2. `parse_node(element)` — crossing 1건 변환

**입력:**
```python
{
  "type": "node",
  "id": 436804138,
  "lat": 37.5445862,
  "lon": 126.9549754,
  "tags": {"highway": "crossing", "crossing": "marked"}
}
```

**변환 로직:**
1. `lat/lon` → `"POINT(lon lat)"`
2. 4개 필드 dict 생성
3. `traffic_signals == "yes"`면 `has_signals=True`

**출력 (4키):**
```python
{
  "osm_id": 436804138,
  "crossing": "marked",
  "has_signals": False,
  "geom_wkt": "POINT(126.9549754 37.5445862)"
}
```

**`parse_response` 반환값:**
```python
(
  [way_dict_1, way_dict_2, ..., way_dict_6290],   # ways: 6,290건
  [cross_dict_1, ..., cross_dict_337]              # crossings: 337건
)
```

### 2-5. ways INSERT — `insert_ways_batch(conn, ways)`

**호출:** `main.py` → `loader/db.py:insert_ways_batch()`

**핵심 로직:**
```python
for chunk in _chunked(ways, 500):  # 500건씩 끊기
    params = [
        (w["osm_id"], w["highway"], w["name"], w["geom_wkt"], ...)
        for w in chunk
    ]
    cur.executemany(_INSERT_WAY, params)
    conn.commit()
```

**전송되는 SQL (`_INSERT_WAY`):**
```sql
INSERT INTO osm_ways (
    osm_id, highway, name, geom,
    foot, access, oneway,
    bridge, tunnel, indoor, layer,
    surface, lit, sidewalk
) VALUES (
    %s, %s, %s, ST_GeomFromText(%s, 4326),
    %s, %s, %s,
    %s, %s, %s, %s,
    %s, %s, %s
)
ON CONFLICT (osm_id) DO NOTHING
```

**파라미터 바인딩 예시 (한 row):**
```
%s 자리에 들어가는 값:
  osm_id    = 26268425
  highway   = "residential"
  name      = None
  geom_wkt  = "LINESTRING(126.9444697 37.5386282, 126.9443664 37.538715)"
  foot      = None
  access    = None
  oneway    = None
  bridge    = False
  tunnel    = False
  indoor    = False
  layer     = 0
  surface   = None
  lit       = None
  sidewalk  = None
```

**PostGIS 처리 흐름:**
1. `geom_wkt` 문자열이 `ST_GeomFromText(%s, 4326)` 함수로 들어감
2. PostGIS가 WKT 문자열을 파싱해 LineString geometry 객체 생성
3. SRID 4326(WGS84)을 부여
4. `osm_ways.geom` 컬럼 제약(`GEOMETRY(LINESTRING, 4326)`)과 매칭 통과
5. 행 INSERT
6. `idx_osm_ways_geom`(GIST), `idx_osm_ways_geom_geog`(GIST geography), `idx_osm_ways_highway`(B-tree)에 인덱스 엔트리 자동 추가

**ON CONFLICT 동작:**
- `osm_id`에 UNIQUE 제약이 있음 (`osm_ways_osm_id_key`)
- 이미 존재하는 osm_id면 "DO NOTHING" → 그 row만 스킵, 트랜잭션은 정상 진행
- 첫 적재에선 모두 신규 INSERT, 재실행 시 모두 스킵

**batch별 commit:**
- 500건 INSERT 후 즉시 commit → 디스크에 영구 반영
- 13개 batch (500 × 12 + 290 × 1) 처리

**반환값:** 처리 시도한 행 수 = 6,290 (실제 적재 건수는 별도 카운트)

**적재 후 DB 상태:**
```
osm_ways: 6,290 행 신규 추가
```

### 2-6. crossings INSERT — `insert_crossings_batch(conn, crossings)`

**호출:** `main.py` → `loader/db.py:insert_crossings_batch()`

**전송되는 SQL (`_INSERT_CROSSING`):**
```sql
INSERT INTO osm_crossings (osm_id, crossing, has_signals, geom)
VALUES (%s, %s, %s, ST_GeomFromText(%s, 4326))
ON CONFLICT (osm_id) DO NOTHING
```

**파라미터 예시:**
```
osm_id      = 436804138
crossing    = "marked"
has_signals = False
geom_wkt    = "POINT(126.9549754 37.5445862)"
```

**PostGIS 처리:** way와 동일한 패턴이지만 LINESTRING이 아닌 POINT geometry. `osm_crossings.geom` 컬럼 제약(`GEOMETRY(POINT, 4326)`) 매칭.

**batch:** 337건이 단일 batch(500 미만)로 처리 → 1회 commit.

**적재 후 DB 상태:**
```
osm_crossings: 337 행 신규 추가
```

### 2-7. 적재 결과 카운트 — `_count()`

**호출:** `main.py` 내부의 private 함수.

**코드:**
```python
def _count(conn, table: str) -> int:
    with conn.cursor() as cur:
        cur.execute(f"SELECT COUNT(*) FROM {table}")
        return cur.fetchone()[0]
```

**전송되는 SQL:**
```sql
SELECT COUNT(*) FROM osm_ways;       -- 6290
SELECT COUNT(*) FROM osm_crossings;  -- 337
```

**왜 INSERT 함수의 반환값을 안 쓰고 다시 SELECT 했나:**
- `insert_*_batch`의 반환값은 "처리 시도 건수"이지 "실제 적재 건수"가 아님
- ON CONFLICT DO NOTHING으로 일부가 스킵될 수 있어 정확한 적재 결과는 SELECT COUNT(*)이 신뢰 가능
- 첫 적재에선 시도 건수 = 적재 건수가 같지만, 재실행 시 0건 적재(전부 스킵)임을 정확히 보여줌

**최종 로그:**
```
2026-04-27 15:55:07,530 [INFO] __main__: 적재 완료: osm_ways=6290, osm_crossings=337
```

### 2-8. 자원 정리 — `finally: conn.close()`

```python
try:
    ... 적재 흐름 ...
finally:
    conn.close()
```

성공이든 실패든 connection을 반드시 닫음. 예외가 나도 connection leak 없음.

## 3. 한눈에 보는 데이터 변환 체인

```
[Overpass 서버]
      │
      │ HTTPS POST (QUERY 본문)
      ▼
fetch_overpass()
      │  반환: dict (JSON 응답)
      │  {"elements": [{"type": "way", "id": ..., "geometry": [...], "tags": {...}}, ...]}
      ▼
parse_response(data)
      │  순회 → parse_way() / parse_node()
      │  반환: (ways: list[dict], crossings: list[dict])
      │
      │  way dict 예:
      │    {"osm_id": 26268425, "geom_wkt": "LINESTRING(126.94 37.53, ...)", ...}
      │  crossing dict 예:
      │    {"osm_id": 436804138, "geom_wkt": "POINT(126.95 37.54)", ...}
      ▼
insert_ways_batch / insert_crossings_batch
      │  500건씩 chunk
      │  executemany(INSERT ... ST_GeomFromText(%s, 4326) ... ON CONFLICT DO NOTHING)
      │  conn.commit()
      ▼
[PostgreSQL + PostGIS]
      │  ST_GeomFromText 가 WKT 문자열을 geometry 객체로 변환
      │  osm_ways.geom (LINESTRING, SRID 4326)
      │  osm_crossings.geom (POINT, SRID 4326)
      │  GIST/B-tree 인덱스 자동 갱신
      ▼
[디스크 영구 저장]  ← osm_ways 6,290행, osm_crossings 337행
```

## 4. 좌표 순서가 뒤집히지 않는 이유 (한 단계 추적)

OSM 좌표가 한국 → 사할린/태평양으로 찍히지 않으려면 lon/lat 순서가 일관돼야 한다. 코드 상에서 이 순서가 어디서 어떻게 보존되는지 추적:

| 단계 | 좌표 형식 | 위치 |
|------|----------|------|
| Overpass 응답 | `{"lat": 37.53, "lon": 126.94}` (키 이름으로 구분) | `fetch_overpass()` 반환값 |
| parse_way 변환 | `f"{p['lon']} {p['lat']}"` → `"126.94 37.53"` | parser.py:24 |
| WKT 문자열 | `LINESTRING(126.94 37.53, ...)` (lon lat 순) | parser.py:25 |
| psycopg2 파라미터 | 문자열 그대로 전달 | db.py executemany |
| PostgreSQL SQL | `ST_GeomFromText('LINESTRING(126.94 37.53, ...)', 4326)` | db.py `_INSERT_WAY` |
| PostGIS 내부 | x=126.94 (경도), y=37.53 (위도) | PostGIS 표준 |
| 컬럼 저장 | geometry 바이너리 (WKB) + SRID 4326 | osm_ways.geom |

키 이름(`lat`/`lon`)을 명시적으로 참조해 직접 직렬화하기 때문에 순서가 깨질 여지가 없다. 만약 Overpass가 키 순서를 바꿔도(예: `{"lon": ..., "lat": ...}`) 결과는 동일하다.

## 5. 멱등성 — 재실행해도 안전한 이유

`python -m loader.main`을 두 번 실행해도 DB는 망가지지 않는다.

| 단계 | 1차 실행 | 2차 실행 |
|------|---------|---------|
| Overpass 호출 | 6,627 element 응답 | 6,627 element 응답 (동일) |
| 파싱 | 6,290 ways + 337 crossings | 동일 |
| INSERT 시도 | 모두 신규 → 6,290 + 337 적재 | 모두 osm_id 충돌 → ON CONFLICT DO NOTHING으로 전부 스킵 |
| `_count` 결과 | 6,290 / 337 | 6,290 / 337 (변화 없음) |

OSM 원본이 갱신되어 새 way가 생기면 그 신규 osm_id만 INSERT되고 기존은 그대로다. 즉 "이미 있는 건 무시, 새 건 추가" 자동.

## 7. 파일별 책임 한 줄 요약

| 파일 | 책임 |
