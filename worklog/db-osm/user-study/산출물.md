좋은 질문입니다. 두 가지 모두 정확히 짚고 가는 게 좋습니다.

## 1. DB에 적재된 것 — 정확히 무엇인가

### `osm_ways` (6,290 행)

각 행은 **"도보가 허용되는 도로 세그먼트 1개"**의 LineString + 메타데이터.

"도보 가능"의 정의는 작업지시 4번 쿼리 조건의 합집합:

| 조건 | 의미 |
|---|---|
| `highway IN (footway, pedestrian, path, steps, residential, ...)` | 적재 대상 도로 분류에 속함 |
| `foot != "no"` | 보행이 명시적으로 금지되지 않음 |
| `access != "private"`, `access != "no"` | 사적 통제가 걸리지 않음 |
| (제외) `motorway`, `trunk` | 자동차 전용 고속도로/간선 제외 |

즉 **"도보 전용 길"이 아니라 "도보가 허용되는 길"**. 분포를 보면:

```
footway     2,043   ← 보행 전용 (인도, 산책로)
service     1,842   ← 골목, 진입로
residential 1,300   ← 주거지 도로
primary       192   ← 큰 차도 (인도 동반 가정)
secondary     126
...
```

`primary/secondary/tertiary`(차도)도 포함된 이유는 한국 도시 환경에서 큰 도로 옆에 인도가 같이 있기 때문. 러닝 앱이 "이 GPS 좌표가 도로망 위에 있는가"를 판단할 때 인도 위만 따로 판별하기 어려우므로, 차도 자체를 보행 가능으로 취급하고 사이드워크 정보(`sidewalk` 컬럼)를 보조로 둔 것.

### `osm_crossings` (337 행)

## 2. LineString은 OSM이 직접 준 건가? — 아니, 가공된 결과

### OSM 원본 데이터 모델

OSM은 LineString을 직접 저장하지 않는다. 원본은 다음 3가지 요소로만 구성:

| 요소 | 정체 |
|---|---|
| **node** | 좌표 점 1개 (`{id, lat, lon}`) |
| **way** | **노드 ID의 순서 있는 리스트** (`{id, nodes: [n1, n2, n3, ...]}`) |
| **relation** | way나 node를 묶은 그룹 (예: 행정 경계, 노선) |

즉 OSM 원본에서 way 26268425는 이렇게 생겼다:

```json
{
  "type": "way",
  "id": 26268425,
  "nodes": [292193234, 292193235, 292193236, ...],  ← 좌표 없음, ID만
  "tags": {"highway": "residential"}
}
```

좌표를 알려면 별도로 node 292193234, 292193235, ...를 각각 조회해서 lat/lon을 가져와 연결해야 한다.

### Overpass API가 좌표를 채워준 이유

쿼리에 `out tags geom;`이라고 적었기 때문이다.

| Overpass 옵션 | 결과 |
|---|---|
| `out;` | way 원본 그대로 (노드 ID 리스트만) |
| `out geom;` | way의 노드들의 좌표를 **함께 펼쳐서 반환** ← 우리가 쓴 옵션 |
| `out tags geom;` | 거기다 모든 tag까지 |

`out geom`을 지정하면 Overpass 서버가 내부적으로 way의 노드들을 dereference해서 좌표 배열을 채워준다:

```json
{
  "type": "way",
  "id": 26268425,
  "geometry": [                          ← Overpass가 채워준 부분
    {"lat": 37.5386282, "lon": 126.9444697},
    {"lat": 37.538715,  "lon": 126.9443664},
    ...
  ],
  "tags": {"highway": "residential", ...}
}
```

이 점에서도 아직 LineString은 아니다. 단순히 좌표 객체의 배열.

### LineString은 우리가 만든 결과

`loader/parser.py:25`에서 좌표 배열을 PostGIS WKT 형식 문자열로 직렬화하는 게 LineString을 만드는 단계:

```python
coords = ", ".join(f"{p['lon']} {p['lat']}" for p in geometry)
geom_wkt = f"LINESTRING({coords})"
# → "LINESTRING(126.9444697 37.5386282, 126.9443664 37.538715, ...)"
```

그리고 PostGIS에 INSERT할 때 `ST_GeomFromText(%s, 4326)`이 이 WKT 문자열을 PostGIS 내부 binary geometry 객체로 변환해 컬럼에 저장. **이 시점에야 비로소 "LineString"이라는 자료형이 된다.**

## 3. 한 줄 정리

| 단계 | 형태 | 만든 주체 |
|---|---|---|
| OSM 원본 | way = 노드 ID 리스트 | OSM 기여자(매핑한 사람) |
| Overpass 응답 | `geometry: [{lat,lon}, ...]` 배열 | Overpass 서버가 노드 dereference |
| 파서 출력 | `"LINESTRING(lon lat, ...)"` WKT 문자열 | 우리 코드 (parser.py) |
| DB 저장 | PostGIS LineString geometry (binary + SRID 4326) | PostGIS의 ST_GeomFromText |

OSM이 LineString 자체를 주는 게 아니라, **OSM의 노드+way 모델을 Overpass가 좌표 배열로 풀어주고, 우리가 PostGIS 형식으로 직렬화**한 결과가 DB에 들어간 것.

면접에서 "OSM 데이터를 어떻게 가져왔나"를 묻는다면:
> "OSM 원본은 노드 ID 리스트 형태의 way라 좌표가 따로 있었습니다. Overpass API의 `out geom` 옵션으로 노드 좌표를 펼쳐 받고, 파서에서 좌표 순서를 (lon, lat)으로 맞춰 WKT LineString 문자열로 직렬화한 다음, PostGIS의 ST_GeomFromText로 SRID 4326 geometry로 저장했습니다."

이렇게 답할 수 있는 게 이번 작업의 학습 포인트 중 하나입니다.
