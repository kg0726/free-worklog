### 구현 내용

**왜 main.py 흐름을 5단계로 잡았나:**
- config 로드 → DB 연결 → fetch → parse → insert는 단방향 데이터 흐름. 단계별 실패가 위쪽 단계로 영향을 주지 않게 순서대로 배치.
- 적재 후 SELECT COUNT(*)를 마지막에 한 번 더 — `insert_*_batch`의 반환값은 "시도 건수"라 실제 적재 건수(ON CONFLICT로 스킵된 건 제외)는 별도 카운트가 정확.

**왜 try/finally로 conn 정리만 하고 예외는 안 잡았나:**
- 적재 중 예외가 나면 stack trace 그대로 노출되어야 디버깅 가능. 잡아서 print하면 원인 정보 손실.
- conn 자원만 finally로 닫음 → 예외 발생 시에도 connection leak 없음.

**왜 02_schema.sql에 GIST geography 함수 인덱스를 추가했나:**
- 검증 쿼리 3번 EXPLAIN에서 `Seq Scan` 발견. 작업지시 8번 완료기준 위반.
- 원인: `ST_DWithin(geom::geography, ..., meters)` 형태는 geometry → geography 캐스팅이 필요. 기존 `idx_osm_ways_geom`(geometry용 GIST)은 캐스팅된 식에 못 쓰임.
- 해결: 캐스팅된 식 자체에 GIST 함수 인덱스 부착 → `CREATE INDEX ... ON osm_ways USING GIST((geom::geography))`. PostGIS에서 미터 단위 거리 쿼리를 인덱스로 가속하는 표준 패턴.
- 인덱스 추가 후 EXPLAIN 재검증: `Seq Scan` → `Index Scan using idx_osm_ways_geom_geog`, 실행 시간 54.787ms → 8.524ms (~6.4배).
- crossings도 같은 패턴이 필요해서 `idx_osm_crossings_geom_geog`도 함께 추가.

**왜 평면 GIST(`idx_osm_ways_geom`)를 그대로 두었나:**
- `ST_Intersects`, `ST_Contains` 같은 평면 연산은 geometry 그대로 사용. 지운 두 인덱스를 모두 유지하는 게 다양한 공간 쿼리 패턴에 대응 가능.
- 인덱스 두 개에 의한 적재 시 비용은 마포구 6,290건 규모에서 무시 가능 (적재 약 2.5초).

### 확인된 이슈 / 결정 사항

#### 이슈: 첫 EXPLAIN에서 Seq Scan 발생 (작업지시 8번 완료기준 위반)
- **증상**: 03_verify.sql 3번 쿼리(`ST_DWithin(geom::geography, point::geography, 1000)`)에서 풀스캔 후 필터링.
- **원인**: 기존 `idx_osm_ways_geom` GIST는 geometry 컬럼에 직접 걸린 인덱스. 쿼리가 `geom::geography`로 캐스팅하면 다른 식이 되므로 인덱스 미사용.
- **해결**: `CREATE INDEX ... USING GIST((geom::geography))`로 캐스팅된 식 자체에 함수 인덱스 추가. EXPLAIN 재실행 시 `Index Scan using idx_osm_ways_geom_geog` 확인.
- **교훈**:
  - PostGIS에서 거리 쿼리(meters)는 geography 타입을 쓰는 것이 정확하지만, geometry 컬럼이 표준이므로 캐스팅 인덱스 패턴이 자주 등장.
  - 평면 연산(geometry: ST_Intersects 등) + 미터 거리 연산(geography) 둘 다 쓸 가능성이 있다면 두 인덱스 다 두는 게 일반적.
  - 작업지시 3번에는 평면 GIST만 명시되어 있었는데, 검증 쿼리(7번)가 geography 캐스팅을 사용 → 둘이 정합되지 않은 작업지시 결함이었고 본 worklog에서 보정.

#### 결정: 02_schema.sql을 수정해 인덱스를 본 스키마의 일부로 포함
- 별도 마이그레이션 파일(예: 04_add_geog_index.sql) 분리도 고려 가능.
- 그러나 MVP 단계라 스키마 본체 + 인덱스를 한 파일에서 보는 편이 후속 작업자가 전체 구조를 빠르게 파악하기 좋음.
- 멱등성(`IF NOT EXISTS`)으로 재실행 안전.
