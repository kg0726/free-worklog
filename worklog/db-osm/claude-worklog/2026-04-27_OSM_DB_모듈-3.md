### 구현 내용

**왜 config.py와 db.py를 분리했나:**
- config는 외부 입력(환경변수) 로드만 담당, db는 PostgreSQL과의 상호작용만 담당. 책임 분리로 테스트·교체가 쉬워짐.
- 운영 적재 시 config만 바꿔 끼우면 동일 db 모듈로 원격 DB에 적재 가능.

**왜 dataclass가 아닌 dict를 반환했나:**
- 반환값을 곧바로 `psycopg2.connect(**cfg)`에 unpacking할 수 있어 변환 코드가 0줄.
- dataclass로 만들면 그 자체로 또 변환 함수가 필요해짐. MVP 규모에서는 dict가 더 단순.

**왜 .env 누락 시 KeyError로 실패하게 두었나:**
- 환경변수는 시스템 boundary(외부 입력). 누락은 명시적 실패가 안전. 기본값으로 살리면 잘못된 DB에 연결되어 silent corruption 위험.
- `os.environ["KEY"]`가 자연스럽게 KeyError를 던지므로 별도 검증 코드 불필요.

**왜 ST_GeomFromText에 SRID 4326을 명시했나:**
- 테이블의 geom 컬럼이 `GEOMETRY(LINESTRING, 4326)`으로 SRID 제약. SRID 없는 geometry를 넣으면 PostGIS가 거부.
- WKT 문자열만으로는 SRID 정보가 없으므로 함수 인자로 명시해야 함.

**왜 ON CONFLICT (osm_id) DO NOTHING 인가:**
- OSM 적재는 멱등이 핵심. 재실행해도 같은 osm_id가 중복 적재되면 안 됨.
- DO UPDATE도 가능하나 OSM 데이터의 갱신 빈도가 낮고 MVP는 1회성 적재라 DO NOTHING으로 충분.

**왜 batch별 commit 인가:**
- 전체 데이터(예: 마포구 way 약 1만 건)를 한 트랜잭션으로 묶으면 중간 실패 시 모두 롤백되어 처음부터 다시 받아야 함.
- ON CONFLICT DO NOTHING과 batch commit 조합으로 "중간에 끊겨도 멱등 재실행"이 보장됨.

**왜 cur.rowcount 대신 처리(시도) 건수를 반환했나:**
- psycopg2의 executemany는 cur.rowcount 동작이 일관적이지 않음 (마지막 statement만 반영되거나 -1 반환되는 케이스).
- 시도 건수는 항상 정확하고, ON CONFLICT로 스킵된 실제 갯수는 적재 후 `SELECT COUNT(*)`로 확인 가능 (C6에서 수행).

**왜 14컬럼 INSERT 컬럼 순서를 그룹핑해서 적었나:**
- 컬럼이 많아 한 줄에 다 적으면 검토가 어려움. (식별/통행제어/구조속성/UX) 4그룹으로 나눠 가독성 확보.
- 파라미터 tuple 생성도 같은 그룹 순서로 정렬해 1:1 대응이 시각적으로 확인됨.

### 주요 함수 / 클래스

#### `load_db_config() -> dict`
- **위치:** `loader/config.py`
- **역할:** 프로젝트 루트의 `.env`를 읽어 psycopg2.connect용 dict를 반환.
- **파라미터:** 없음
- **리턴:** `{"host": str, "port": int, "dbname": str, "user": str, "password": str}` — psycopg2.connect()에 그대로 unpacking 가능
- **주의사항 / 사이드이펙트:**
  - `dotenv.load_dotenv()`가 os.environ을 변경 (글로벌 사이드이펙트)
  - 환경변수 5개 중 하나라도 누락 시 KeyError
  - port는 int로 변환 (환경변수는 항상 문자열이므로)

#### `get_connection(cfg: dict)`
- **위치:** `loader/db.py`
- **역할:** psycopg2 connection 객체 반환.
- **파라미터:**
  - `cfg`: `load_db_config()` 반환 dict
- **리턴:** `psycopg2.extensions.connection`
- **주의사항:** 호출자가 close 책임. main에서는 컨텍스트 매니저나 finally로 닫아야 함.

#### `insert_ways_batch(conn, ways: list[dict], batch_size: int = 500) -> int`
- **위치:** `loader/db.py`
- **역할:** ways를 batch_size씩 끊어 osm_ways에 INSERT (ON CONFLICT DO NOTHING).
- **파라미터:**
  - `conn`: psycopg2 connection
  - `ways`: parser.parse_way 출력 dict 리스트. 각 dict는 14개 키 필요: `osm_id, highway, name, geom_wkt, foot, access, oneway, bridge, tunnel, indoor, layer, surface, lit, sidewalk`
  - `batch_size`: 한 번에 executemany로 보낼 건수 (기본 500)
- **리턴:** 처리 시도한 행 수 (스킵 포함). 실제 적재 건수는 SELECT COUNT(*)로 확인.
- **주의사항 / 사이드이펙트:**
  - 각 batch마다 commit. 중간 실패 시 직전 batch까지 적재됨.
  - geom_wkt는 `LINESTRING(lon lat, lon lat ...)` 문자열. ST_GeomFromText(%s, 4326)로 변환됨.
  - dict에 키가 누락되면 KeyError. parser 단계에서 None을 명시적으로 채워야 함.

#### `insert_crossings_batch(conn, crossings: list[dict], batch_size: int = 500) -> int`
- **위치:** `loader/db.py`
- **역할:** crossings를 batch_size씩 끊어 osm_crossings에 INSERT (ON CONFLICT DO NOTHING).
- **파라미터:**
  - `conn`: psycopg2 connection
  - `crossings`: parser.parse_node 출력 dict 리스트. 각 dict는 4개 키: `osm_id, crossing, has_signals, geom_wkt`
  - `batch_size`: 기본 500
- **리턴:** 처리 시도한 행 수.
- **주의사항:** geom_wkt는 `POINT(lon lat)` 문자열.

#### `_chunked(items: list, size: int) -> Iterator[list]`
- **위치:** `loader/db.py` (private)
- **역할:** 리스트를 size 단위로 끊어주는 제너레이터.
- **파라미터:** items, size
- **리턴:** 리스트 슬라이스 반복자

### 확인된 이슈 / 결정 사항
- **executemany rowcount 신뢰 안 함**: psycopg2 executemany는 rowcount 동작이 환경에 따라 달라 정확한 적재 건수 측정용으로 부적합. 시도 건수만 반환하고 실제 적재 건수는 SELECT COUNT(*)로 확인하기로 결정.
- **PostgreSQL 16.4 (Debian) 확인**: 컨테이너 내부 PostgreSQL 버전 16.4. PostGIS 3.4와 정상 매칭.
- **psycopg2 vs psycopg3**: 작업지시가 psycopg2-binary로 명시. psycopg3가 더 신문법(async, connection pool 내장)이지만 Hibernate 사용하는 Spring 백엔드와 별개 적재 스크립트라 검증된 psycopg2로 충분.
