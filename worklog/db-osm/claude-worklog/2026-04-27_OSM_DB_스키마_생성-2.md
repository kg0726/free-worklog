## [2026-04-27] OSM DB 스키마 생성

### 작업 범위

- `자율/data/osm-loader/sql/01_extensions.sql` 신규 생성
- `자율/data/osm-loader/sql/02_schema.sql` 신규 생성
- `자율/data/osm-loader/sql/03_verify.sql` 신규 생성
- `localPostgis` 컨테이너에 01, 02 적용 → `osm_ways`, `osm_crossings` 테이블 + 인덱스 생성됨

### 구현 내용

**왜 SQL 파일을 분리했나:**

- `01_extensions.sql`: PostGIS extension 활성화. 컨테이너 이미지(`postgis/postgis:16-3.4`)는 init 시 자동으로 PostGIS를 만들어두지만, 다른 환경(맨 PostgreSQL 이미지) 적용도 고려해 `IF NOT EXISTS`로 안전하게 추가.
- `02_schema.sql`: 테이블·인덱스 DDL. 본 작업의 핵심 산출물.
- `03_verify.sql`: 적재 후(C6) 결과 검증용. 미리 만들어두면 매번 쿼리를 다시 짤 필요 없이 재실행 가능.

**왜 `osm_ways`와 `osm_crossings`를 분리했나:**

- 한 테이블에 way(LINESTRING)와 crossing(POINT)을 함께 두려면 geometry 타입을 일반화(GEOMETRY)해야 하고, 이러면 컬럼·인덱스 의미가 흐려짐.
- 분리하면 각 테이블이 단일 geometry 타입으로 명확해지고, 공간 인덱스도 타입별로 최적화됨.

**왜 SRID 4326 (WGS84) 인가:**

- GPS와 OSM이 모두 WGS84 좌표계(위경도)를 기본으로 사용. 다른 좌표계로 저장하면 적재 시 변환 비용이 들고, GPS 좌표 검증 시 매번 ST_Transform이 필요해짐.
- 거리 계산이 필요한 곳에서는 `geom::geography`로 캐스팅해 미터 단위 계산.

**왜 GIST 공간 인덱스인가:**

- ST_DWithin·ST_Intersects 등 공간 쿼리는 일반 B-tree로 가속 불가. GIST는 R-tree 계열로 2D bounding box를 트리화해 공간 쿼리에 최적.
- 이 인덱스가 없으면 마포구 전체 스캔이 발생해 검증 로직이 100ms → 10s 단위로 느려짐.

**왜 highway에 별도 B-tree 인덱스를 추가했나:**

- "footway만 추출", "primary/secondary 제외" 같은 도로 분류 필터링이 검증 로직에서 자주 발생함.
- highway는 약 14개 카테고리로 카디널리티가 낮지만, 각 분류가 명확히 구분되므로 B-tree로 충분히 가속됨.

**왜 `bridge`/`tunnel`/`indoor`를 BOOLEAN으로 풀어두었나:**

- OSM 원본 태그는 `bridge=yes/no/viaduct/aqueduct/...` 다양한 값을 가지지만, MVP 검증 로직에서는 "GPS가 끊겨도 면책"이라는 boolean 판단만 필요.
- 파서에서 `tags.get("bridge") == "yes"`로 단순화. 추후 세부 분류가 필요해지면 컬럼 추가로 확장 가능.

**왜 `lit`만 nullable BOOLEAN인가 (다른 boolean은 DEFAULT FALSE):**

- 야간 안전 경로 추천 기능에서 "조명 정보 없음"과 "조명 없음"이 구별되어야 함. 정보 없음을 FALSE로 처리하면 OSM 미태깅 도로가 모두 야간 위험으로 분류됨.
- `bridge`/`tunnel`/`indoor`는 미태깅 = 일반 도로(false)가 자연스러우므로 DEFAULT FALSE.

**`03_verify.sql`에 ST_IsValid 검증을 추가한 이유:**

- 작업지시 8번 완료 기준의 "geometry 좌표 순서 오류 없음" 항목을 자동화. 좌표를 (lon, lat) 대신 (lat, lon)으로 잘못 넣으면 ST_IsValid는 통과해도 좌표가 한국 → 사할린·태평양 어딘가로 찍힘. 그러나 좌표 자체가 깨진 경우(중복 점, 자기 교차 등)는 ST_IsValid가 false로 잡아냄.
- 실행 결과 0이면 모든 geometry가 PostGIS 관점에서 유효.

### DB 변경 사항

#### `osm_ways` (신규 테이블)

| 컬럼           | 타입                           | 제약                                 | 설명                                      |
| ------------ | ---------------------------- | ---------------------------------- | --------------------------------------- |
| `id`         | `BIGSERIAL`                  | PK                                 | 내부 PK, 자동 증가                            |
| `osm_id`     | `BIGINT`                     | NOT NULL UNIQUE                    | OSM 전역 고유 way ID. ON CONFLICT 기준 키      |
| `highway`    | `VARCHAR(50)`                | NOT NULL                           | 도로 분류 (footway/residential/primary 등)   |
| `name`       | `VARCHAR(255)`               | NULL 허용                            | 도로명 (없는 도로 다수)                          |
| `geom`       | `GEOMETRY(LINESTRING, 4326)` | NOT NULL                           | WGS84 좌표계의 도로 선형                        |
| `foot`       | `VARCHAR(20)`                | NULL                               | 보행 허용 여부 (yes/no/designated/permissive) |
| `access`     | `VARCHAR(20)`                | NULL                               | 일반 출입 통제 (yes/no/private/customers)     |
| `oneway`     | `VARCHAR(10)`                | NULL                               | 일방통행 (yes/no/-1)                        |
| `bridge`     | `BOOLEAN`                    | DEFAULT FALSE                      | 다리 위 — GPS 끊김 면책용                       |
| `tunnel`     | `BOOLEAN`                    | DEFAULT FALSE                      | 터널 — GPS 끊김 면책용                         |
| `indoor`     | `BOOLEAN`                    | DEFAULT FALSE                      | 실내 통로 — GPS 끊김 면책용                      |
| `layer`      | `SMALLINT`                   | DEFAULT 0                          | 입체 교차 층 (-2 ~ 2 등)                      |
| `surface`    | `VARCHAR(30)`                | NULL                               | 노면 (asphalt/concrete/dirt 등). 러닝 UX 추천용 |
| `lit`        | `BOOLEAN`                    | NULL 허용                            | 조명. NULL = 정보 없음, FALSE = 명시적 없음        |
| `sidewalk`   | `VARCHAR(20)`                | NULL                               | 인도 위치 (both/left/right/none)            |
| `created_at` | `TIMESTAMP`                  | NOT NULL DEFAULT CURRENT_TIMESTAMP | 적재 시각                                   |

- **인덱스:**
  - `osm_ways_pkey` (PK, btree on `id`)
  - `osm_ways_osm_id_key` (UNIQUE on `osm_id`) — ON CONFLICT (osm_id) 동작에 사용
  - `idx_osm_ways_geom` (GIST on `geom`) — 공간 쿼리 가속 (ST_DWithin/ST_Intersects)
  - `idx_osm_ways_highway` (btree on `highway`) — 도로 분류 필터링
- **연관관계:** 기존 ERD의 `nodes` 테이블(사용자 GPS 경로용)과 무관. 완전히 분리된 별도 테이블.
- **주의사항:**
  - `geom` SRID는 반드시 4326. 적재 시 `ST_GeomFromText('LINESTRING(...)', 4326)` 사용.
  - 좌표 순서는 `lon lat` (PostGIS WKT 표준). Overpass API는 `{"lat":y, "lon":x}`로 반환하므로 파서에서 순서 변환 필수.

#### `osm_crossings` (신규 테이블)

| 컬럼            | 타입                      | 제약                                 | 설명                                              |
| ------------- | ----------------------- | ---------------------------------- | ----------------------------------------------- |
| `id`          | `BIGSERIAL`             | PK                                 | 내부 PK                                           |
| `osm_id`      | `BIGINT`                | NOT NULL UNIQUE                    | OSM 전역 고유 node ID                               |
| `crossing`    | `VARCHAR(30)`           | NULL                               | 횡단보도 종류 (marked/unmarked/zebra/traffic_signals) |
| `has_signals` | `BOOLEAN`               | DEFAULT FALSE                      | 신호등 유무 (`traffic_signals` 태그 = "yes")           |
| `geom`        | `GEOMETRY(POINT, 4326)` | NOT NULL                           | 횡단보도 좌표 (단일 점)                                  |
| `created_at`  | `TIMESTAMP`             | NOT NULL DEFAULT CURRENT_TIMESTAMP | 적재 시각                                           |

- **인덱스:**
  - `osm_crossings_pkey` (PK, btree on `id`)
  - `osm_crossings_osm_id_key` (UNIQUE on `osm_id`) — ON CONFLICT 동작
  - `idx_osm_crossings_geom` (GIST on `geom`) — 공간 쿼리
- **연관관계:** 논리적으로 `osm_ways`와 가까이 위치(공간적으로 이웃)할 수 있으나 FK 없음. 검증 로직에서 `ST_DWithin(crossing.geom, way.geom, 5)`로 동적 매칭.
- **주의사항:**
  - 작업지시 3번 비고대로 한국 OSM의 `highway=crossing` 커버리지가 낮아 MVP 검증 로직은 1순위 사용처가 아님. 보조 데이터로만 활용 예정.

### 확인된 이슈 / 결정 사항

- **PostGIS extension 이미 존재**: `postgis/postgis:16-3.4` 이미지가 init 시 자동 생성. `IF NOT EXISTS`로 멱등 처리됨(NOTICE만 출력하고 정상 종료).
- **created_at 타입을 TIMESTAMP(timezone 없음)로 한 점**: 작업지시 3번 DDL을 그대로 따름. Spring Boot가 PostgreSQL 적재 데이터를 읽을 때 `LocalDateTime`으로 매핑됨. 운영 시 timezone 문제가 생기면 `TIMESTAMPTZ`로 변경 검토 필요(미결).
- **인덱스 명 일관성**: 작업지시의 `idx_osm_ways_geom`, `idx_osm_ways_highway`, `idx_osm_crossings_geom` 명명 그대로 사용. UNIQUE 제약은 PostgreSQL이 자동으로 `<table>_<column>_key` 형식 인덱스를 생성함(`osm_ways_osm_id_key`).
