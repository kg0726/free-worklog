### 구현 내용

**왜 파서를 별도 모듈로 분리했나:**
- HTTP 응답 형식(Overpass)과 DB INSERT 형식(osm_ways/osm_crossings 컬럼)을 잇는 변환 계층. 둘 중 하나가 바뀌어도 파서만 수정하면 됨.
- DB 모듈(`db.py`)은 dict in / SQL out, 클라이언트 모듈은 HTTP / dict out — 파서가 가운데에서 형식 매핑만 담당.

**왜 way의 geometry < 2점은 None을 반환하나:**
- PostGIS `LINESTRING`은 최소 2점 필요. 1점짜리 LineString을 INSERT하면 PostGIS가 거부하고 트랜잭션 실패.
- Overpass에서 `out tags geom`으로 받았을 때 area 경계나 데이터 결손으로 geometry가 빈 way가 드물게 등장 가능. 적재 단계가 아닌 파싱 단계에서 거르면 db 모듈이 단순해짐.
- None 반환 → 호출자(`parse_response`)에서 None 필터링 + 카운트 로깅. 어떤 way가 왜 빠졌는지 추적 가능.

**왜 좌표 순서를 `lon lat`로 직렬화했나:**
- PostGIS WKT 표준 좌표 순서가 `(x y)` = `(lon lat)`. WGS84(SRID 4326)에서 x=경도, y=위도.
- Overpass 응답은 `{"lat": y, "lon": x}` 객체. 키 이름만 보고 그대로 직렬화하면 좌표가 뒤집힘.
- 좌표 순서 오류는 `ST_IsValid` 검증으로 못 잡힘 — 좌표 자체는 valid한 LineString이지만 사할린·태평양 같은 엉뚱한 위치로 찍힘. 파싱 단계에서 한 번 명확히 결정하는 것이 가장 안전.

**왜 layer는 try/except로 0 폴백 처리했나:**
- 작업지시 5번 표가 `int(tags.get("layer", 0))` + "실패 시 0"을 명시. OSM 태그값은 사용자 입력이라 `"1.5"`, `"-1.0"`, `"abc"` 같은 비정수 가능성이 0이 아님.
- 0으로 폴백 = "지상층"으로 간주. 입체 교차로 정보가 깨졌을 때 무시하는 게 적재 실패보다 나음 (MVP 검증 로직은 layer를 직접 안 씀, 인접 way 연결성 판단 보조).

**왜 lit는 3-state(None/True/False)로 다뤘나:**
- DB 테이블에서 `lit`만 nullable BOOLEAN으로 설계됨 (worklog C2 참조).
- 의미: NULL = 정보 없음, TRUE = 조명 명시, FALSE = 조명 없음 명시.
- 파싱 시 `tags.get("lit") is None`이면 NULL, 그 외에는 `=="yes"`로 boolean 변환. 작업지시 5번 표 그대로.
- 야간 안전 경로 추천 시 NULL을 FALSE로 간주하면 미태깅 도로가 모두 위험하게 분류됨 — 정보 없음과 명시적 없음 구분이 중요한 이유.

**왜 다른 boolean들(bridge/tunnel/indoor/has_signals)은 단순 `== "yes"`로 처리했나:**
- DB에서 DEFAULT FALSE로 설계됨. 미태깅 = 일반 도로/일반 교차로로 간주.
- OSM에서 `bridge=viaduct/aqueduct/...` 같은 세부 분류도 존재하지만 MVP 검증 로직은 "다리 위인가?" boolean만 필요.
- `=="yes"`만 통과 → 세부 분류는 모두 FALSE로 묶임. 향후 세분 처리 필요 시 컬럼 추가 + 파싱 분기 추가로 확장.

**왜 `parse_response`가 두 리스트 분리 + None 필터링까지 한 번에 하는가:**
- main에서 사용할 때 elements를 한 번만 순회하면 됨. way/node 분리·필터링·로깅을 분리하면 같은 리스트를 여러 번 순회.
- 6,000건 규모에서 성능 차이는 무시할 수 있지만, 명령형 코드가 한곳에 모이면 "응답 → DB 입력" 흐름 전체가 한눈에 보임.

### 주요 함수 / 클래스

#### `parse_way(element: dict) -> dict | None`
- **위치:** `loader/parser.py`
- **역할:** Overpass way element 1건을 osm_ways INSERT용 14키 dict로 변환.
- **파라미터:**
  - `element`: Overpass JSON 응답의 한 way element. 필요 키: `id`, `geometry`, `tags`.
- **리턴:**
  - geometry < 2점이면 `None` (LINESTRING 불가).
  - 그 외엔 14키 dict: `osm_id, highway, name, geom_wkt, foot, access, oneway, bridge, tunnel, indoor, layer, surface, lit, sidewalk`.
- **주의사항 / 사이드이펙트:**
  - `tags["highway"]` 직접 접근 → 없는 경우 KeyError. Overpass 쿼리가 `way["highway"~...]`로 거른 결과만 들어오므로 항상 존재한다고 가정.
  - geom_wkt는 SRID 없는 순수 WKT 문자열. db 모듈에서 `ST_GeomFromText(%s, 4326)`로 SRID 부여.

#### `parse_node(element: dict) -> dict`
- **위치:** `loader/parser.py`
- **역할:** Overpass node element 1건을 osm_crossings INSERT용 4키 dict로 변환.
- **파라미터:**
  - `element`: Overpass JSON node element. 필요 키: `id`, `lat`, `lon`, `tags`.
- **리턴:** `{"osm_id", "crossing", "has_signals", "geom_wkt"}` 4키 dict.
- **주의사항:**
  - geometry는 단일 점 → `POINT(lon lat)`.
  - geometry 부족 같은 예외 케이스 없음 (node는 항상 lat/lon 보유).

#### `parse_response(data: dict) -> tuple[list[dict], list[dict]]`
- **위치:** `loader/parser.py`
- **역할:** Overpass 응답 전체를 way 리스트와 crossing 리스트로 분리.
- **파라미터:**
  - `data`: `fetch_overpass()` 반환 dict. `data["elements"]`에 way/node 혼합.
- **리턴:** `(ways, crossings)` 튜플. ways는 `parse_way`가 None을 반환한 항목 제외.
- **주의사항 / 사이드이펙트:**
  - 제거된 way 건수와 최종 way/crossing 카운트를 logger.info로 출력.
  - way/node 외 type(예: relation)은 무시.

### 확인된 이슈 / 결정 사항

#### 결정: way 키 누락 시 KeyError 허용
- `tags["highway"]`는 Overpass 쿼리가 highway 필터를 적용한 결과라 항상 존재한다고 가정.
- 그 외 14필드 중 nullable인 필드는 `tags.get(...)`으로 None 처리. 누락이 정상이라는 의미.
- 만약 `tags["highway"]`가 빠진 element가 들어오면 KeyError로 빠르게 실패 → 쿼리 또는 응답 형식 자체에 이상이 있다는 뜻이므로 silent skip보다 명시적 실패가 안전.

#### 결정: relation 등 다른 type 무시
- 작업지시 4번 쿼리는 way + node만 요청하지만 향후 area를 추가하면 다른 type이 섞일 수 있음.
- `if/elif`로 way/node만 처리하고 나머지는 silent skip. 명시적 type 판별이라 의도가 분명.

#### 콘솔 한글 출력 깨짐 (cp949)
- Windows cmd 기본 인코딩 cp949 문제. Python 내부 문자열은 utf-8로 정상이고, DB INSERT 시에도 utf-8로 전달되므로 적재 결과에는 영향 없음. 검증 시 `psql`로 SELECT 결과를 보면 정상 출력될 것 (그건 컨테이너 내부에서 utf-8).
